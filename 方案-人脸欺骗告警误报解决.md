# 人脸欺骗告警（Spoof/Attack）误报解决方案

> 问题：正常的人脸检测频繁触发"人脸欺骗告警"，导致大量误报。
> 日期：2026-07-14

---

## 一、根因分析

经过全面审查活体检测与报警链路的每一环，定位到以下 **5 个根因**：

### 🔴 根因 1：报警触发是「单帧决策」—— 最关键问题

- **位置**：`pipeline.py` 第 659-682 行
- **机制**：当前代码只要 `face_recognizer` 返回 `"Spoof/Attack"`，就立即推入 `alarm_queue` 触发报警。
- **问题**：
  - 与正常识别结果（走 `temporal_filter` 15 帧滑动窗口 + 快慢通道共识）不同，Spoof 报警**完全跳过了时序滤波**，一帧决定生死。
  - 只要某一帧因光照、角度、轻微遮挡等原因活体检测分数低于 0.1，即触发报警。
  - `triggered_spoofs` 集合只防重复推送，但一旦追踪丢失再重获（检测框抖动导致 tracker 短暂丢ID），会**重新报警**。

### 🔴 根因 2：框运动惩罚（Penalty_box）过于敏感

- **位置**：`liveness_detector.py` 第 123-125 行
- **当前值**：当 `area_ratio < 0.8 or area_ratio > 1.25 or norm_dist > 0.15` 时，`Penalty_box = 0.2`
- **问题**：
  - 正常人走路时，YOLO 检测框会因姿态变化自然波动 10-30%，跑步或快走时波动更大。
  - 这个 0.2 的惩罚对分数影响极大。假设一个真实人脸的正常综合分数为 0.25（处于 `Unknown` 区间），被扣 0.2 后降到 0.05 → **直接进入 Spoof 区间**。
  - 更进一步：框抖动本身只是在**相邻帧之间**判断，如果一帧因为姿态变化框变小了 → 面积比突然增大 → 触发惩罚 → Spoof。

### 🔴 根因 3：决策阈值区间过窄

- **位置**：`liveness_detector.py` 第 133/141 行
- **当前值**：
  - `Score >= 0.3` → **Live**
  - `Score <= 0.1` → **Spoof**
  - 中间 → **Unknown**
- **问题**：
  - Live 与 Spoof 之间只有 0.2 宽的缓冲区。
  - MiniFASNetV2 模型在新场景（不同摄像头、光照、角度）下的输出可能系统性偏低，真实人脸的 `P_fas` 可能只有 0.3-0.6。
  - 当真实人脸的 `P_fas = 0.4`，加上清晰度 (`Var_lap_norm`) 和 HSV (`Std_hsv_norm`) 各得 0.4，综合分数 = `0.6×0.4 + 0.2×0.4 + 0.2×0.4 = 0.40`。如果没有框运动惩罚勉强 Live，若被扣 0.2 → 0.20 → 进入 Unknown 但不触发报警。**但是**如果清晰度或颜色分布也较低（实际场景中很常见），分数进一步下降，再叠加惩罚，很容易低于 0.1。

### 🔴 根因 4：白名单 TTL 过短

- **位置**：`liveness_detector.py` 第 136 行
- **当前值**：`"expire": now + 3.0`（3 秒）
- **问题**：
  - 3 秒过期后需要重新计算活体，重新进入决策区间。
  - 如果一个人站在摄像头前超过 3 秒，活体缓存反复失效 → 重新计算 → 可能在某些帧落入 Spoof。

### 🔴 根因 5：Laplacian 清晰度 + HSV 方差不适用于远距离/低分辨率场景

- **位置**：`liveness_detector.py` 第 98-109 行
- **机制**：
  - `Var_lap_norm = min(1.0, laplacian_var / 300.0)`：拉普拉斯方差 300 以上才算 1.0
  - HSV 方差使用固定归一化分母（`std_s / 20.0`, `std_v / 30.0`）
- **问题**：
  - 远距离人脸（例如安防场景 5-10 米外）裁剪后只有几十像素，Laplacian 方差自然很低（20-50），归一化后只有 0.07-0.17。
  - 低分辨率人脸的 HSV 方差也较低（颜色信息少），两个辅助指标几乎失效。
  - 此时综合分数几乎完全依赖 MiniFASNetV2 模型输出，如果模型输出也偏低，极易落入 Spoof。

## 二、解决方案

### 方案一：增加 Spoof 的时序共识机制（最高优先级，必做）

**目标**：让 Spoof 报警走与正常识别相同的时序滤波逻辑，而不是单帧决策。

**改动点**：`pipeline.py` 第 659-682 行 + 新增 `spoof_temporal_filter`

**具体设计**：

```
为每个 obj_id 维护一个 spoof_temporal_filter，记录最近 N 帧的活体检测状态：
  - "live"     → 活体检测结果为 Live
  - "spoof"    → 活体检测结果为 Spoof
  - "unknown"  → 活体检测结果为 Unknown

报警触发策略（二选一）：
  A.【推荐】连续 Spoof 策略：连续 M 帧（M=5）都判定为 Spoof 才触发报警
  B. 多数投票策略：最近 15 帧中 Spoof 占比 >= 60% 才触发报警
```

**推荐采用策略 A（连续 Spoof）**，原因：
- 正常人脸即使偶尔一帧因噪声被判 Spoof，下一帧大概率会恢复为 Live 或 Unknown，不会达到连续阈值
- 真正的欺骗攻击（照片/屏幕翻拍）会持续返回 Spoof
- 实现简单，逻辑清晰

```python
# ===== 新增：spoof_temporal_filter =====
# 在 __init__ 中：
self.spoof_temporal_filter = {}  # { obj_id: deque(maxlen=15) }

# 在 run_inference 中，修改第659-682行：
if name == "Spoof/Attack":
    if obj_id not in self.spoof_temporal_filter:
        self.spoof_temporal_filter[obj_id] = []
    self.spoof_temporal_filter[obj_id].append("spoof")
    if len(self.spoof_temporal_filter[obj_id]) > 15:
        self.spoof_temporal_filter[obj_id].pop(0)
    
    # 连续5帧判定为Spoof才触发报警
    hist = self.spoof_temporal_filter[obj_id]
    if len(hist) >= 5 and all(s == "spoof" for s in hist[-5:]):
        if obj_id not in self.triggered_spoofs:
            self.triggered_spoofs.add(obj_id)
            # → 原有报警逻辑（推送 alarm_queue）
        # 否则：不触发报警，继续观察
    consensus_name = "Spoof/Attack"
else:
    # 如果当前帧不是Spoof，清空该ID的spoof时序记录
    self.spoof_temporal_filter.pop(obj_id, None)
    # ... 原有 temporal_filter 逻辑不变
```

**同时**：在 cleanup 中同步清理 `spoof_temporal_filter`：
```python
# 在约第531-536行新增
self.spoof_temporal_filter.pop(active_id, None)
```

### 方案二：降低框运动惩罚敏感度（高优先级）

**目标**：减少正常运动姿态变化导致的误判。

**改动点**：`liveness_detector.py` 第 123-125 行

**修改方案**：

| 参数 | 当前值 | 建议值 | 原因 |
|------|--------|--------|------|
| 面积比下限 | < 0.8 | < 0.6 | 正常行走框变化通常不超过 40% |
| 面积比上限 | > 1.25 | > 1.6 | 同上 |
| 中心距离阈值 | > 0.15 | > 0.25 | 宽裕度提升约 66% |
| 惩罚值 | 0.2 | 0.1 | 减半，降低对综合分数的影响 |

**可选增强**：对人脸检测置信度进行加权，置信度高的帧惩罚减弱 50%。

### 方案三：优化决策阈值（中优先级）

**目标**：扩大 Unknown 缓冲区间，避免边缘值误报。

**改动点**：`liveness_detector.py` 第 133/141 行

**修改方案**：

| 阈值 | 当前值 | 建议值 | 原因 |
|------|--------|--------|------|
| Live 阈值 | >= 0.3 | >= 0.25 | 略微放宽，给真实人脸更大余量 |
| Spoof 阈值 | <= 0.1 | <= 0.05 | 大幅收紧，只有极低分数才判定为欺骗 |

**效果**：Unknown 区间从 [0.1, 0.3) 扩大到 [0.05, 0.25)，缓冲区增大 2 倍。

### 方案四：延长白名单 TTL + 自适应续期（中优先级）

**目标**：减少同一目标反复重新计算活体的频率。

**改动点**：`liveness_detector.py` 第 136 行

**修改方案**：

```python
# 白名单 TTL 从 3 秒延长到 8 秒
"expire": now + 8.0

# 新增：每次检测到同一 Live 目标时自动续期（最多 30 秒）
if cached and now <= cached.get("expire", 0):
    # 如果还处于白名单有效期内，每次命中自动续期 3 秒（上限 30 秒）
    remaining = cached["expire"] - now
    if remaining < 25.0:  # 避免无限制延长
        cached["expire"] = now + min(remaining + 3.0, 30.0)
```

### 方案五：分级报警策略（低优先级）

**目标**：降低误报的感知影响，即使触发也分级别处理。

**改动点**：`pipeline.py` 报警推送逻辑

**设计**：

```python
# 根据连续 spoof 帧数决定报警级别
continuous_spoof_count = len(hist)  # hist 为 spoof_temporal_filter 中的记录

if continuous_spoof_count >= 15:
    alarm_level = "critical"   # 持续欺骗 → 严重
elif continuous_spoof_count >= 10:
    alarm_level = "high"       # 较长时间欺骗 → 高
elif continuous_spoof_count >= 5:
    alarm_level = "medium"     # 连续5帧 → 中等（第一级报警）
```

**可选**：
- medium 级别报警仅在系统内记录，**不触发钉钉通知**
- high/critical 级别才触发钉钉通知
- 这样可以大幅降低钉钉的误报频次

### 方案六：增加活体检测场景自适应校准（低优先级，可选）

**目标**：根据不同摄像头、光照条件自适应调整活体检测参数。

**改动点**：`liveness_detector.py` + 新增配置

**设计**：

```python
# 在 LivenessDetector 初始化时，根据摄像头ID加载场景参数
# 配置文件或数据库存储每个摄像头的推荐参数

# 自适应基线采集（启动后空闲时自动采集 30 帧建立基线）：
# 1. 统计正常场景下的平均 Laplacian 方差
# 2. 统计正常场景下的平均 HSV 标准差
# 3. 根据基线值动态调整 Laplacian 和 HSV 的归一化分母
```

## 四、风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| Spoof 时序共识导致真正的欺骗攻击延迟报警（最多 5 帧 ≈ 0.2-0.5 秒）| 低 | 5帧延迟对安防场景可接受，远低于人工响应时间 |
| Spoof 阈值降低 (0.1→0.05) 可能漏过部分攻击 | 中 | 低值 Spoof 的真实攻击极少具有如此高的活体分数；核心防伪能力仍由 MiniFASNetV2 模型保障 |
| 白名单 TTL 延长可能导致真正攻击有更长的"假活体"窗口 | 低 | 白名单仅在 `Score >= 0.25` 时设置，真正的攻击要达到此分数极难 |

## 五、验证建议

1. **回放测试**：用历史误报的视频片段回放，验证 Phase 1 的修改是否消除了误报
2. **攻击测试**：用手机屏幕、打印照片等真实攻击手段验证 Spoof 检测能力未下降
3. **长时间跑测**：在真实场景下连续运行 24 小时，统计误报率变化
4. **回归测试**：确保围栏入侵、跌倒、火情等其他报警类型不受影响
